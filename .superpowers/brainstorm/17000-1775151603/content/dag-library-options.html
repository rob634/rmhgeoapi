DAG Visualization Library Options

Each renders your process_raster workflow left-to-right. Consider: complexity, dependency weight, visual quality.

A

Pure SVG + CSS (No Library)

Server-side Python computes Sugiyama-style rank assignment. Outputs positioned SVG boxes and path edges. CSS handles colors/status. Zero JS dependencies.

process_raster — Pure SVG

Pros

  • Zero JS dependencies
  • Server-side layout = no FOUC
  • Full control over rendering
  • Works in preview server

Cons

  • Must write rank/position algorithm
  • Edge routing is manual work
  • No automatic crossing minimization
B

Dagre (JS Sugiyama layout)

Server provides node/edge JSON. Dagre computes positions client-side. We render to SVG manually (no dagre-d3 needed — just dagre for layout math). ~30KB gzipped.

process_raster — Dagre Layout → SVG

Pros

  • Automatic Sugiyama layout (rank, order, position)
  • Handles edge crossing minimization
  • We only write SVG rendering, not layout math
  • Proven algorithm, widely used

Cons

  • ~30KB JS dependency (CDN or vendored)
  • Client-side layout = brief flash before render
  • Dagre is unmaintained (last release 2019, but stable)
C

ELK.js (Eclipse Layout Kernel)

Industrial-strength layout engine from Eclipse. Handles hierarchical layout, edge routing, port constraints. Used by VS Code's debug visualizer. ~180KB gzipped.

process_raster — ELK Layout

Same visual output as B — ELK produces slightly more polished edge routing but the difference is minimal for DAGs of this size.

ELK shines on DAGs with 50+ nodes and complex port constraints. Your largest workflow (process_raster_collection) has 19 nodes.

Pros

  • Best layout quality (orthogonal edge routing)
  • Actively maintained
  • Handles very complex graphs

Cons

  • 180KB — 6x heavier than dagre
  • Async API (WASM-based)
  • Overkill for 5-19 node DAGs
  • More complex integration

Recommendation: B (Dagre)

Sweet spot — automatic layout without writing rank assignment code, small footprint, proven stable. "Unmaintained" is fine here because Sugiyama is a solved algorithm — dagre does one thing and does it correctly. Your largest workflow is 19 nodes; dagre handles hundreds. Option A means writing your own layout engine. Option C is overkill.
