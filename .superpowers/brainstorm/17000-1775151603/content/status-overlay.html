Status Overlay on Nodes

When viewing a specific run, how should node status (completed, running, failed, etc.) appear on the diagram?

A

Fill Color = Status (type via shape only)

Node fill color changes to reflect runtime status. The shape alone carries the type. This is the classic approach — green=done, blue=running, red=failed, gray=pending.

process_raster — mid-execution (validate running)

Pros

  • Status is the most prominent visual signal
  • Instantly see where the workflow is at
  • Shape still communicates node type

Cons

  • Loses the type-specific color coding from the YAML-only view
B

Border + Badge = Status (fill keeps type color)

Node fill stays the type color (blue for task, amber for conditional, etc). A thick border color and small icon badge show status. Preserves both type AND status.

process_raster — mid-execution (validate running)

Pros

  • Preserves BOTH type color and status
  • Consistent with YAML-only view (same fill colors)
  • Badge icons are unambiguous

Cons

  • More visual noise (two color channels + badge)
  • Pending nodes are harder to spot — subtle border change
  • Status less immediately obvious than full-fill approach

Recommendation: A (Fill = Status)

When viewing a live run, the #1 question is "where is it and what broke?" — status matters more than type. The shapes already carry type information (diamond = conditional, octagon = gate). Double-encoding type via shape AND color is redundant when status is the thing you need to see instantly. The YAML-only definition view (no run context) can keep the type colors since there's no status to show.
